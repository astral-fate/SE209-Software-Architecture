What is the SDLC?

The Software Development Life Cycle (SDLC) is the process used by the software industry to design, develop, and test high-quality software.

It starts after a business analyst gathers requirements and writes a Software Requirement Specification (SRS) document. The development team then follows a series of sequential steps.

Where Does Architecture Fit In?

The Software Architecture is defined and documented during the Design phase of the SDLC.

üèõÔ∏è What is Software Architecture?

The Blueprint of a System

Architecture serves as a blueprint for a system. It provides an abstraction to manage the system's complexity and establishes a communication and coordination mechanism among components.

It defines a structured solution to meet all technical and operational requirements, while optimizing for quality attributes like performance and security.

Why is Architecture Important?

  • It enables the selection of Quality Attributes (like scalability, reliability) to be optimized.
  • It helps everyone better understand the system, which reduces development cost.
  • It facilitates early prototyping to monitor customer needs.
  • It allows the system to be built in component-wise, enabling teams to work on individual pieces in isolation.

What's in a Good Architectural Model?

A good architecture defines:

  • The structure and behavior of a system.
  • Component relationships and communication structure.
  • A balance of stakeholders' needs.
  • It also influences team structure and captures early design decisions.

‚ú® Quality Attributes (The "ilities")

Software architecture enables the selection of Quality Attributes to be optimized for a system. These are also known as non-functional requirements.

Scalability ‚ñº

A system's capacity to manage an expanding workload without compromising performance. (Page 8)

Reliability ‚ñº

The ability of a system to carry out its intended functions under predefined conditions for a certain time without interruption. (Page 8)

Availability ‚ñº

A system's capacity to respond quickly and easily to user requests. Uses techniques like load balancing, redundancy, and failover. (Page 9)

Maintainability ‚ñº

A system's ease of modification or updating. Aided by modularity and low coupling. (Page 9)

Security ‚ñº

A system's capacity to fend off illegal entry and data breaches. Uses access control, authentication, and encryption. (Page 10)

Performance ‚ñº

The measure of how quickly and effectively a system operates. Uses caching, indexing, and query optimization. (Page 10)

Testability ‚ñº

A system's capacity for efficient testing (unit testing, integration testing, etc.). (Page 11)

Usability ‚ñº

How well users can perform a specific task on the system (e.g., registering an account). (Page 11)

Portability ‚ñº

The ability to be easily transferred or adapted to different platforms, operating systems, or environments. (Page 12)

Interoperability ‚ñº

The ability of different software systems to communicate, exchange data, and work together seamlessly. (Page 12)

Compatibility ‚ñº

The degree to which a system can exchange data or perform functions while sharing the same hardware or software environment. (Page 13)

Flexibility ‚ñº

A system's capacity to adjust to shifting demands, including extensibility and configurability. (Page 13)

üß© Styles vs. Patterns vs. Design Patterns

The Hierarchy of Design

It's important to understand the difference between these three related concepts. Think of it as a hierarchy from the highest level (Style) to the lowest level (Design Pattern).

Architecture Style (e.g., Microservices)

The 10,000-foot helicopter view. Shows the highest level of abstraction, how code is organized, and supports certain quality attributes.

Examples: Layered, Client-Server, Microservices, Event-Driven.

Architecture Pattern (e.g., CQRS)

A solution to a recurring problem *within* an architectural style. Describes how components interact.

Examples: MVC, MVP, MVVM, N-Tier.

What are Design Patterns?

Design Patterns are the most granular. They are best practices and experiences used to solve common, general problems *during* development.

  • Creational: Provide ways to create objects while hiding the logic (e.g., Factory, Builder).
  • Structural: Concerned with class and object composition.
  • Behavioral: Concerned with communication between objects.

üèóÔ∏è Architecture Types: Monolithic vs. Distributed

The Two Main Classifications

Architectural styles can be broadly classified by their deployment structure:

  • Monolithic (Single Deployment Unit): The application is created as a single entity, sharing the same resources and memory.
  • Distributed (Multiple Deployment Units): Consists of multiple services that work together.

Monolithic vs. Distributed

üëç Pros of Monolithic

  • Much simpler to design and implement.
  • Inexpensive (lower initial cost).
  • Can be developed and deployed much more quickly.
  • Suitable for small to medium-sized apps with simple functionality.

üëé Cons of Monolithic

  • Maintenance is challenging (any update requires redeploying the *entire* app).
  • Limited scalability (the *whole* application must be scaled at once).
  • Mean Time to Recovery (MTTR) is high (minutes). A fatal error causes *all* functionality to fail.

üëç Pros of Distributed

  • Scalability is at the individual service level.
  • High reliability and fault tolerance (if one service fails, others can continue).
  • Low MTTR (seconds or milliseconds). Failed services can recover quickly.
  • High flexibility and testability (changes are isolated to a single service).

üëé Cons of Distributed

  • Higher cost (initial implementation and ongoing maintenance).
  • More complex to design and manage.
  • Subject to the "8 Fallacies of Distributed Computing" (e.g., believing the network is always reliable, latency is zero, etc.).

üß± Architecture Partitioning

How is the System's Structure Organized?

Architectures can also be classified by how the system is partitioned. This choice MUST be aligned with your team structure and the expected types of changes.

üíª

Technical Partitioning

Components are organized by technical usage (e.g., Presentation, Business, Persistence layers). All code for a single feature (like "customer") is spread across all layers.

Namespace: app.presentation.customer, app.business.customer

Use Case: Good if changes are isolated to a technical layer (e.g., changing the UI look and feel, swapping a database).

Team Structure: Teams of UI developers, backend developers, and DBAs.

üì¶

Domain Partitioning

Components are organized by domain areas (e.g., Customer, Shipping, Payment). All code for a single feature (UI, business, data) is grouped together in one place.

Namespace: app.customer.presentation, app.customer.business

Use Case: Good if changes are isolated to a domain (e.g., adding a feature to the "wish list"). This is much more effective for domain-scoped changes.

Team Structure: Cross-functional teams (UI, backend, DBA all on one team).
